# 01. Концепция проекта

## 1. Назначение документа

Этот документ фиксирует первую часть проектирования будущего веб-проекта для косметологического салона / салона красоты.

Документ описывает не техническую реализацию, а продуктовую концепцию: какую задачу решает проект, каким должен быть пользовательский опыт, какие принципы должны сохраняться при дальнейшей разработке и какие решения уже были приняты.

Этот файл можно использовать как основу для дальнейшей работы над проектом и как ориентир при выборе технологий, проектировании интерфейса, написании кода и создании архитектуры.

---

## 2. Общая идея проекта

Проект представляет собой современную веб-платформу для салона красоты и косметологических услуг.

Это не просто сайт-визитка. Проект должен объединять:

- презентацию салона;
- каталог услуг;
- страницы мастеров;
- онлайн-запись;
- личный кабинет клиента;
- систему отзывов;
- полезные материалы;
- уведомления;
- дальнейшую возможность расширения до внутренней системы для мастеров и администрации.

Основная идея проекта — создать не шаблонный сайт, а кастомизированный продукт, который выгодно отличается от типичных решений на конструкторах.

Главная цель — сделать интерфейс спокойным, понятным, полезным и не навязчивым.

---

## 3. Философия проекта

Главная философия проекта:

**Сайт должен заботиться о пользователе еще до того, как он стал клиентом салона.**

Это означает:

- не давить регистрацией;
- не навязывать запись с первых секунд;
- не перегружать информацией;
- не использовать агрессивные цвета и раздражающие элементы;
- не заставлять пользователя принимать лишние решения;
- давать информацию постепенно;
- помогать пользователю спокойно прийти к решению.

Сайт должен создавать ощущение:

- спокойствия;
- расслабленности;
- доверия;
- чистоты;
- профессионализма;
- заботы;
- премиальности без вычурности.

Один из главных образов проекта:  
**пользователь должен ощущать сайт так, будто ему уже делают мягкий расслабляющий массаж.**

---

## 4. Целевая аудитория

Проект рассчитан на широкую целевую аудиторию.

Основная аудитория — взрослые мужчины и женщины, заинтересованные в услугах салона красоты, ухода за внешностью, парикмахерских услугах, маникюре, косметологических процедурах и смежных направлениях.

При этом проект не должен жестко ограничиваться только взрослой аудиторией. В будущем салон может предоставлять отдельные услуги детям или подросткам, если это будет уместно.

Важно не сужать аудиторию искусственно. Сайт должен быть понятным, спокойным и доступным для разных категорий пользователей.

---

## 5. Основные направления услуг

Проект может включать разные категории услуг:

- парикмахерские услуги;
- маникюр и педикюр;
- косметологические процедуры;
- уходовые процедуры;
- массаж лица;
- более сложные косметологические направления с медицинскими особенностями.

При этом важно не перегружать пользователя сразу всеми деталями. На первом уровне пользователь должен увидеть общую картину, а уже потом углубляться в конкретные процедуры.

---

## 6. Основные принципы UX

### 6.1. Не навязывать действия

Пользователь не должен сразу получать требование зарегистрироваться или записаться.

Сначала он знакомится с сайтом, услугами, атмосферой салона и только потом, когда он уже выбрал услугу и время, система предлагает пройти регистрацию.

### 6.2. Не перегружать информацией

Главная страница не должна рассказывать абсолютно всё.

Ее задача — заинтересовать и помочь пользователю перейти дальше.

Внутренние страницы должны содержать только ту информацию, которая помогает принять конкретное решение.

### 6.3. Каждая страница имеет свою роль

Главная страница — режим знакомства.  
Страница услуги — режим выбора.  
Страница мастера — режим доверия к специалисту.  
Процесс записи — режим быстрого действия.  
Личный кабинет — режим полезной информации.

### 6.4. Минимизировать количество действий

Пользователь должен принимать только те решения, которые действительно необходимы:

- выбрать услугу;
- выбрать мастера;
- выбрать дату;
- выбрать время;
- подтвердить запись.

Все остальное система должна делать сама.

### 6.5. Не добавлять сложность без необходимости

Функциональность должна появляться только тогда, когда она реально полезна.

Например, если мастеров всего 2–3, фильтры не нужны.  
Если мастеров 20–30, фильтры становятся полезными.

### 6.6. Декор не конкурирует с контентом

Фон, анимации, текстуры, золотые акценты и визуальные эффекты должны усиливать атмосферу, но не отвлекать пользователя от основной информации.

---

## 7. Пользовательские роли

На первом этапе проектирования были определены следующие роли:

### 7.1. Гость

Гость — незарегистрированный пользователь.

Он может:

- смотреть главную страницу;
- изучать услуги;
- смотреть мастеров;
- смотреть галерею;
- читать полезные материалы;
- начинать процесс записи.

Гость не должен сталкиваться с регистрацией до момента, когда он действительно хочет выполнить действие, связанное с персональными данными или записью.

### 7.2. Клиент

Клиент — зарегистрированный пользователь.

Он может:

- записываться на услуги;
- просматривать активные записи;
- получать уведомления;
- переносить или отменять запись;
- смотреть историю посещений;
- оставлять отзывы;
- управлять своими контактными данными.

### 7.3. Мастер

Роль мастера заложена в будущий этап проекта, но пока не проектируется подробно.

В будущем у мастера должен быть собственный dashboard.

### 7.4. Администратор

Администратор пока не является фокусом первой версии клиентской части.

Роль можно заложить архитектурно, но детально проектировать позже.

### 7.5. Владелец

Владелец — роль с максимальными правами в системе.

Подробная логика владельца будет проектироваться на этапе внутренней системы.

---

## 8. Разделение этапов проекта

Принято важное решение:  
**сначала проектируется и реализуется клиентская часть.**

Только после завершения клиентского интерфейса планируется переход к:

- dashboard мастера;
- админке;
- CRM-элементам;
- внутренней коммуникации;
- голосовым заметкам;
- управлению расписанием;
- аналитике.

Такой подход позволяет не распыляться и довести одну часть проекта до логического завершения.

---

## 9. Главная страница

### 9.1. Назначение

Главная страница — это не каталог и не прайс-лист.

Ее задача:

- создать первое впечатление;
- показать атмосферу салона;
- вызвать доверие;
- дать общее понимание услуг;
- мягко провести пользователя дальше.

Главная страница должна работать как спокойный лендинг.

### 9.2. Первый экран

Первый экран должен занимать почти всю высоту экрана.

Основной акцент — не на тексте, а на атмосфере:

- фотография входа;
- интерьер салона;
- спокойная презентационная картинка;
- плавная смена 3–5 изображений.

Анимация должна быть мягкой, без резких переходов. Возможен эффект легкого белого размытия при смене кадра.

### 9.3. Navbar

Навигационная панель должна быть динамической:

- на первом экране она видна поверх изображения;
- при скролле вниз плавно скрывается;
- при легком скролле вверх снова появляется;
- не должна закрывать контент и отвлекать пользователя.

### 9.4. Цветовая концепция

Основное направление:

**белая основа с благородными золотыми акцентами.**

Золото должно быть именно акцентом, а не доминирующим цветом.

Общее ощущение:

- светло;
- мягко;
- дорого, но не вычурно;
- спокойно;
- чисто.

### 9.5. Структура главной страницы

Ориентировочная последовательность блоков:

1. Hero-блок.
2. Краткая навигация по возможностям салона.
3. Более подробное знакомство с направлениями услуг.
4. Блок доверия: почему нам доверяют.
5. Небольшой блок отзывов.
6. Финальный спокойный призыв к действию.
7. Футер.

### 9.6. Цены на главной странице

На главной странице не нужно делать акцент на ценах.

Цены не должны отпугивать пользователя на этапе первого знакомства.

При этом цены не должны быть скрыты полностью. Они могут отображаться на страницах конкретных услуг.

### 9.7. Финальный CTA

Финальный призыв к действию уместен именно в конце страницы.

Он не воспринимается как давление, потому что пользователь уже познакомился с сайтом.

Пример:

- записаться онлайн;
- получить консультацию.

### 9.8. Футер

Футер должен содержать:

- контакты;
- мессенджеры;
- Telegram;
- WhatsApp;
- Viber;
- адрес;
- график работы;
- ссылку на карту;
- юридические ссылки при необходимости.

---

## 10. Визуальные переходы и декоративные элементы

Между смысловыми блоками можно использовать мягкие декоративные переходы.

Идея:

- не вставлять яркие фотографии;
- использовать ненавязчивый фон;
- легкие текстуры;
- мягкие обои;
- светлые декоративные элементы.

Такие переходы должны создавать ощущение воздуха и плавности, но не отвлекать.

Главное правило:

**Декор никогда не должен конкурировать с содержанием.**

---

## 11. Mobile First и адаптация

Проект должен одинаково хорошо работать:

- на десктопе;
- на ноутбуке;
- на планшете;
- на телефоне.

Особое внимание — мобильной версии.

Пользователь может открыть сайт:

- вечером;
- ночью;
- на улице;
- на ярком солнце;
- на ходу.

Поэтому интерфейс должен учитывать:

- читаемость;
- контрастность;
- крупные элементы;
- удобство нажатия большим пальцем;
- возможность темной темы;
- возможность ручного переключения темы.

Автоматическая тема может ориентироваться на настройки устройства, но ручной выбор должен быть возможен.

---

## 12. Страница конкретной услуги

### 12.1. Назначение

Страница услуги отличается от главной.

Главная страница — режим знакомства.  
Страница услуги — режим выбора.

Пользователь уже пришел за конкретной информацией, поэтому страница должна быть компактной и не превращаться во второй лендинг.

### 12.2. Первый экран услуги

Можно использовать Hero-блок, связанный с конкретной услугой.

Допустимо 3–5 изображений:

- процедура;
- рабочий процесс;
- кабинет;
- оборудование;
- результат.

### 12.3. Информация на странице услуги

Страница должна отвечать на основные вопросы:

- что это за процедура;
- кому подходит;
- длительность;
- стоимость;
- кто выполняет;
- как записаться;
- важные ограничения или противопоказания в кратком виде.

Подробные материалы лучше выносить в раздел полезной информации.

### 12.4. Не перегружать страницу

Если пользователь хочет глубже изучить тему, ему можно предложить полезные материалы:

- как подготовиться;
- что нужно знать;
- частые вопросы;
- уход после процедуры.

---

## 13. Выбор мастера

После выбора услуги пользователь видит мастеров, которые могут выполнить эту услугу.

### 13.1. Первый уровень выбора

Карточка мастера на этапе выбора должна быть простой:

- фото;
- имя;
- краткая специализация;
- рейтинг;
- ближайшее свободное время, если уместно.

Главный акцент — визуальное первое впечатление.

### 13.2. Подробная карточка мастера

При клике пользователь может открыть подробную информацию:

- опыт;
- специализация;
- сертификаты;
- портфолио;
- отзывы;
- выполненные работы;
- ближайшие даты;
- кнопка записи.

### 13.3. Фильтры

Фильтры не нужны, если мастеров мало.

Если мастеров становится много, можно добавить:

- пол специалиста;
- специализацию;
- рейтинг;
- ближайшее свободное время;
- опыт.

Принцип:

**Функциональность должна масштабироваться вместе с проектом.**

---

## 14. Процесс записи

Основная логика:

**Услуга → Мастер → Дата → Время → Регистрация / авторизация → Подтверждение заявки**

### 14.1. Выбор услуги

Клиент обычно приходит за услугой, а не за конкретным мастером.

Поэтому процесс начинается с выбора услуги.

### 14.2. Выбор мастера

После выбора услуги система показывает мастеров, которые могут ее выполнить.

### 14.3. Выбор даты

Выбор даты должен быть интуитивным.

Решение:

- показывать текущий и следующий месяц;
- не заставлять пользователя выбирать год;
- использовать привычный календарный формат;
- показывать цветовую индикацию занятости.

Возможная логика цветов:

- зеленый — есть свободные окна;
- желтый — осталось мало времени;
- красный — полностью занято;
- серый — нерабочий день.

### 14.4. Выбор времени

После выбора даты появляется табличный блок со свободным временем.

Время не должно быть простым списком.

Причина: табличный формат ближе к привычному восприятию календаря.

Пользователь выбирает конкретное время.  
Длительность процедуры система рассчитывает автоматически.

### 14.5. Подтверждение

Перед финальным шагом клиент видит карточку с деталями:

- услуга;
- мастер;
- дата;
- время;
- длительность;
- стоимость.

---

## 15. Регистрация и авторизация

Регистрация появляется только тогда, когда пользователь уже выбрал услугу, мастера, дату и время.

Это воспринимается не как навязчивость, а как естественный шаг подтверждения записи.

Важно не использовать сухой текст «Регистрация».

Лучше объяснить смысл:

**Последний шаг для подтверждения записи.**

Регистрация нужна для:

- сохранения записи;
- уведомлений;
- возможности переноса или отмены;
- истории посещений;
- отзывов.

Способы авторизации — email, Telegram или другие варианты — будут обсуждаться позже на техническом этапе.

SMS использовать не планируется.

---

## 16. Подтверждение мастером

Запись не считается окончательно подтвержденной сразу после отправки клиентом.

Сначала создается заявка со статусом:

**Ожидает подтверждения мастером.**

Мастер должен подтвердить запись, потому что могут быть реальные причины отказа:

- поломка оборудования;
- изменение графика;
- болезнь;
- отпуск;
- другие обстоятельства.

Клиенту не нужно писать «Вы записаны», пока мастер не подтвердил заявку.

Правильное сообщение:

**Ваша заявка успешно отправлена мастеру. Мы уведомим вас после подтверждения.**

Возможные статусы записи:

- ожидает подтверждения мастером;
- подтверждена;
- отклонена мастером;
- отменена клиентом;
- запрошен перенос;
- завершена.

---

## 17. После отправки заявки

После отправки заявки пользователь видит спокойный экран подтверждения.

Визуальный символ — зеленая галочка.

Зеленый цвет воспринимается как сигнал, что процесс идет правильно.

Сообщение должно быть примерно таким:

**Ваша заявка отправлена. Сейчас она ожидает подтверждения мастером.**

Дальше система уведомляет клиента:

- после подтверждения мастером;
- за сутки до визита;
- за несколько часов до визита, если это будет уместно;
- после визита с предложением оставить отзыв.

---

## 18. Личный кабинет клиента

Личный кабинет должен быть полезным, а не формальным.

Главный принцип:

**Максимум полезной информации, минимум второстепенной.**

### 18.1. Основной экран

На первом месте — активные записи.

Если у клиента есть будущая запись, он сразу должен видеть:

- дату;
- время;
- услугу;
- мастера;
- статус;
- возможность переноса;
- возможность отмены.

### 18.2. Информация о клиенте

Можно показывать:

- как давно человек является клиентом;
- количество посещений;
- историю процедур;
- персональные предложения.

### 18.3. Персональные события

В личном кабинете могут появляться мягкие персональные предложения:

- поздравление с днем рождения;
- скидка на день рождения;
- персональная рекомендация;
- напоминание о повторной процедуре.

Важно не превращать личный кабинет в рекламную витрину.

### 18.4. История посещений

Клиент может видеть прошлые визиты и при желании повторить услугу.

### 18.5. Отзывы

Клиент может видеть:

- какие отзывы он оставил;
- какие визиты еще можно оценить.

---

## 19. Отзывы клиентов

Отзывы являются важной частью доверия.

Оставить отзыв должен иметь возможность только клиент после реального завершенного визита.

Форматы отзыва:

- быстрая оценка;
- лайк;
- короткий комментарий;
- подробный отзыв;
- похвала мастера;
- отзыв о конкретной процедуре.

Положительные отзывы после модерации могут отображаться:

- на главной странице;
- в блоке доверия;
- на странице мастера;
- на странице услуги.

На главной странице отзывы лучше показывать ненавязчиво: небольшой блок или карточка, которая плавно меняется.

Не нужно делать огромную стену отзывов.

Отзывы должны выглядеть естественно, без рекламного тона.

---

## 20. Полезная информация

Раздел не стоит называть «Блог».

Слово «блог» может восприниматься как новости или длинные статьи.

Лучше использовать нейтральное и полезное название:

- полезная информация;
- полезные материалы;
- советы специалистов;
- полезно знать;
- рекомендации по уходу.

Главный принцип:

**Мы не продаем — мы делимся опытом.**

Раздел нужен для пользователей, которые хотят разобраться глубже.

Там могут быть материалы:

- как подготовиться к процедуре;
- уход после процедуры;
- частые вопросы;
- мифы и правда;
- советы специалистов;
- рекомендации по домашнему уходу.

Полезные материалы можно связывать со страницами услуг.

---

## 21. Галерея и контент

Видео в первой версии не используем.

Причина: видео может раздражать, отвлекать и нарушать спокойную атмосферу проекта.

Галерея должна быть отдельным разделом.

Ее задача — не просто показать фотографии, а стать путеводителем по салону.

Галерея может включать:

- интерьер;
- кабинеты;
- оборудование;
- зону ожидания;
- детали атмосферы;
- рабочие места.

Фотографий не должно быть слишком много.

Лучше меньше, но качественнее.

Главное правило:

**Фотография должна помогать, а не развлекать.**

---

## 22. Контакты

Отдельную страницу контактов на первом этапе можно не делать.

Контактную информацию достаточно разместить в футере:

- адрес;
- телефон;
- график работы;
- мессенджеры;
- ссылка на карту;
- построить маршрут.

При необходимости можно сделать кликабельную ссылку, открывающую карту.

---

## 23. Поиск по сайту

Поиск в первой версии не нужен.

Проект не является библиотекой с большим количеством материалов.

Информация должна быть организована так, чтобы пользователь находил все интуитивно через структуру сайта.

Поиск можно добавить позже, если объем контента значительно вырастет.

---

## 24. Решения, которые отложены

На данном этапе не проектируются подробно:

- AI-помощник;
- dashboard мастера;
- админка;
- CRM;
- внутренняя коммуникация персонала;
- голосовые заметки;
- финансовая аналитика;
- сложные фильтры;
- поиск;
- полноценная система акций;
- юридические документы.

Эти темы будут рассмотрены на следующих этапах.

---

## 25. Основные принятые решения

1. Сначала проектируется клиентская часть.
2. Главная страница — спокойный лендинг.
3. Регистрация не появляется в начале.
4. Цены не акцентируются на главной странице.
5. Видео не используются.
6. Поиск не нужен в первой версии.
7. Галерея — отдельный путеводитель по салону.
8. Отзывы показываются ненавязчиво.
9. Запись начинается с выбора услуги.
10. Запись подтверждается мастером.
11. Личный кабинет показывает прежде всего активные записи.
12. Полезная информация заменяет классический блог.
13. Декор не должен отвлекать от контента.
14. Интерфейс должен быть спокойным, понятным и масштабируемым.

---

## 26. Главная идея первой части проекта

Клиентская часть проекта должна быть не просто красивой.

Она должна вести пользователя по естественному пути:

1. заинтересоваться;
2. познакомиться;
3. довериться;
4. выбрать услугу;
5. выбрать мастера;
6. записаться;
7. получить подтверждение;
8. прийти на визит;
9. оставить отзыв;
10. вернуться снова.

Главная ценность проекта — уважение к вниманию пользователя.

Сайт ничего не навязывает, не перегружает и не требует лишнего. Он помогает клиенту спокойно принять решение и пройти путь от первого знакомства до визита в салон.
